# ADR 0020 — Spec text refresh for v2.0 (drop "(1.0)" scope labels, align with reality)

**Status:** Accepted (2026-05-27)
**Spec version:** editorial within 2.0.0 (no normative behavior change)

## 1. Context

A multi-agent audit of spec v2.0.0 surfaced a cluster of stale wordings
inherited from the v1.x source-of-truth:

- `spec/00-overview.md` headings were labelled `In scope (1.0)` /
  `Out of scope (1.0)`, and the "spec version is the shared anchor" line
  pointed at "§7 of the design doc" — a doc deleted in the v1.1 cleanup.
- `spec/03-messages.md` opened the concrete-message section with
  "VMx 1.0 defines two concrete messages."
- `spec/07-group-vm.md` ended its modeled-variant discussion with
  "In v1.0 only the non-modeled variant ships."
- `spec/08-aggregate-vm.md` framed itself around "VMx v1.0 ships arities
  1 through 5" and asserted that aggregate `construct()` / `destruct()`
  drive their slots "in parallel" — every flavor's reference
  implementation actually drives them sequentially.
- `spec/11-threading.md` labelled the TypeScript scheduler row "(future)"
  even though TypeScript shipped at v2.0.0.
- `spec/16-notifications.md` did not say what happens when the same
  `Notification` instance is posted twice — the impls keyed a dictionary
  by identity, so a duplicate post silently dropped the original
  awaitable.
- `spec/ADRs/0002` referenced "§13 of the design doc."
- `spec/ADRs/0006` opened with "VMx ships in C#, Python, and future
  TypeScript."
- `spec/ADRs/0007`'s conformance summary repeated the "parallel"
  language.
- `spec/ADRs/0009` recorded the Python `RelayCommandOfT` /
  `AggregateVMBuilderN` removals as "deferred to v2.0.0 (breaking)" —
  but those aliases still ship in v2.0.0.
- `spec/ADRs/0011`'s TypeScript clause prescribed a `.sources(...)`
  builder taking a rest-parameter list; the shipped TypeScript surface is
  a free function `fromSources(sources, transform, opts?)` (rest + a
  trailing options object would conflict).

None of these change behavior. They are editorial drift between the spec
text and the v2.0 implementations, plus one previously-undefined corner
in the Notifications hub.

## 2. Decision

Apply an editorial refresh that aligns the spec text with the shipped
v2.0 implementations:

### 2.1 Scope-label cleanup

- Drop `(1.0)` / `(future)` qualifiers from chapter headings and the
  threading-scheduler table. Replace prose like "VMx 1.0 / v1.0 ships X"
  with version-agnostic phrasing that reflects what is in the box today.

### 2.2 Aggregate construct/destruct ordering

- `spec/08-aggregate-vm.md` §Construction / §Destruction now read like
  the corresponding sections in `spec/06-composite-vm.md`: the order is
  unspecified and the reference implementations drive the slots
  sequentially. ADR-0007's conformance summary is reworded to match.

### 2.3 Notifications: duplicate Post

- `spec/16-notifications.md` §Post adds a clarifying sentence:
  duplicate-post is implementation-defined; implementations SHOULD
  return the existing awaitable rather than silently dropping it. The
  rule is intentionally SHOULD, not MUST, so the v2.0 hubs are not
  retroactively non-conformant. A future minor version may strengthen
  it to MUST and add a `NOTIF-011` conformance ID.

### 2.4 ADR-0006 / ADR-0009 alignment

- ADR-0006 opens with "C#, Python, and TypeScript" (TypeScript shipped).
- ADR-0009's deferred-rename section moves the Python `RelayCommandOfT`
  and `AggregateVMBuilderN` removals from `v2.0.0` to `v3.0.0`,
  recording that the v2.0.0 cut intentionally preserved the legacy
  aliases for downstream code pinned to v1.x.

### 2.5 ADR-0011 alignment

- ADR-0011's TypeScript clause documents the shipped surface
  (`fromSources(sources, transform, opts?)`) and the reason the
  originally proposed rest-parameter builder was not adopted.

### 2.6 Dead references

- `spec/00-overview.md` and `spec/ADRs/0002` drop the
  "§7 / §13 of the design doc" pointers (the design doc was removed in
  the v1.1 cleanup); the same intent is now expressed by an ADR
  cross-reference where applicable.

## 3. Consequences

- No new conformance ID. The corresponding tests already exist; their
  evidence still holds because no behavior changed.
- No new fixture, no new public API.
- The ADR-0006 / ADR-0009 / ADR-0011 amendments are recorded in this
  ADR (per the spec-discipline workflow, spec changes need an ADR even
  if the change is only editorial).
- Cross-flavor parity is maintained: the spec text now matches the
  shipped C#, Python, and TypeScript implementations rather than the
  v1.x intentions.

## 4. Rejected alternatives

- **Strengthen the duplicate-Post rule to a normative no-op in v2.0.0.**
  Would require touching `NotificationHub` in all three flavors and
  adding `NOTIF-011`; defers cleanly to v2.1 / v3.0 where the change
  is in scope.
- **Promote the aggregate-parallel intent into a real parallel
  implementation.** Possible but expensive (dispatcher choice, error
  aggregation, deterministic shutdown ordering) — not warranted by a
  spec-text refresh. The reference implementations remain sequential.
