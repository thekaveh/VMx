# ADR 0037 — Spec v2.5.0 maintenance clarifications and additions

**Status:** Accepted (2026-06-10)
**Spec version:** introduced in 2.5.0

## 1. Context

An overnight maintenance audit of the four flavors against the spec surfaced a
cluster of issues that need spec-level rulings rather than silent code fixes:
two flavors emitted hub `PropertyChangedMessage` names in mixed casings, the
`Clear()`/`Count` interaction was read two contradictory ways by different
test suites, `ReparentChild` silently corrupted trees on ancestor cycles in
every flavor, `NotificationHub` shutdown semantics existed only in C#, and the
Swift flavor's first release both diverged from the lifecycle raise contract
and overclaimed its conformance subset.

## 2. Decisions

### 2.1 Hub `PropertyName` follows the flavor idiom (clarification)

Chapter 03 §2.1 now states explicitly what the chapter 04 examples implied:
the `PropertyName` payload uses the emitting flavor's naming idiom per
ADR-0006 (`"IsValid"` / `"is_valid"` / `"isValid"`). TypeScript and Swift
emitted PascalCase from some call sites (`"Model"`, `"Current"`,
`"Component1..6"`, `"IsCurrent"`) and camelCase from others; both now emit
camelCase everywhere. This is a conformance fix, not a contract change —
but it is observable by subscribers that filtered on the wrong casing, so it
ships as a minor-version behavior change with changelog entries.

### 2.2 `Clear()` emits `PropertyChanged("Count")` (clarification)

Chapter 21 §3.3 said Count fires "after every mutation that changes Count
(adds and removes, not replace)"; a Python unit test read bulk `Clear()` as
exempt while the batch-exit rule (§3.5) already mandated Count-after-Reset
for count-changing batches. The contradiction is resolved in favor of the
batch-exit precedent: `Clear()` on a non-empty list emits Count after Reset;
clearing an empty list emits nothing. All three full-parity flavors fixed.

### 2.3 `ReparentChild` rejects self- and ancestor-reparenting (HIER-018)

No flavor guarded `ReparentChild` against reparenting a node under itself or
under its own descendant; the resulting parent cycle made `Depth`, `Path`,
and `walk` diverge or recurse indefinitely — the same silent-corruption class
as the v2.4.x `ServicedObservableCollection.Move` fix. Chapter 18 §5 now
mandates rejection with the flavor's standard invalid-operation error, no
structural change, and no message. New catalog ID **HIER-018**; implemented
with conformance tests in C#, Python, and TypeScript (Swift's subset does not
include HIER).

### 2.4 `NotificationHub` dispose semantics (NOTIF-017)

C#'s hub (since the v2.4.x race fixes) resolves in-flight waiters with
`Pending` on dispose, completes the `Pending` observable, refuses new
enqueues, and is idempotent. Python and TypeScript had no dispose at all,
leaking permanently-pending awaitables at shutdown. Chapter 16 §9 now
specifies the C# semantics for every flavor; new catalog ID **NOTIF-017**.
Python additionally adopts the emit-inside-lock discipline from the C# fix.

### 2.5 Swift lifecycle raise is a trap (documented divergence)

Chapter 02 §2 listed Swift among the flavors that raise
`StatusTransitionError`. The Swift flavor traps (`preconditionFailure`)
instead. Rather than force `try` onto every legal lifecycle call site, the
divergence is documented: traps are the Swift-stdlib convention for API
misuse, and the gating predicates (`canConstruct()` etc.) remain the
supported pre-flight check. Consequences:

- Swift cannot claim LIFE-005/006 beyond their gating predicates, cannot
  claim LIFE-008 at all, and COMP-009 is likewise unclaimable.
- A throwing lifecycle variant remains open as a future major if demand
  appears.

### 2.6 Swift conformance subset recount (corrects ADR-0036 §2.E)

ADR-0036 declared a 53-ID Swift subset; auditing the tests against the
catalog found ~30 citations shifted or vacuous, and several claimed IDs
(COMP-001/002, GRP-001/005/006, CMD-005/007, CVM-006, LIFE-008/011) had no
implemented behavior behind them. The honest subset is **39 IDs**:
LIFE-001..007/009/010/012/013, CVM-001..006, COMP-003..005, GRP-002..004,
AGG-001..006, CMD-001..004/006, BLD-001..005 — with LIFE-005/006 asserting
the gates (per §2.5) and CVM-003's read-only contract surfaced as a trap.
The built-in command wiring, readonly builder, and Equatable `==` builder
default that several of these IDs depend on were implemented as part of this
decision. Deferred Swift items: fixture-backed LIFE-011, CollectionChanged
events, AutoConstructOnAdd/BatchUpdate, the parameterized command, and the
truth-table fixture.

> **Forward note (VMX-107).** This 39-ID subset corrects **ADR-0036 §2.E**'s
> "53 of 232". It later grows to **41** when **ADR-0042** adds the
> CompositeVM `current`/`onCurrentChanged` builder IDs **COMP-025/026** to the
> Swift subset — the figure carried forward in `01-concepts.md` §1.

### 2.7 Editorial corrections

- Chapter 15 §7 described the derived-properties fixture with a `sources`
  key (the fixture's key is `sources_initial`) and misdescribed
  `expected_values` (it includes the initial reading, per DPROP-012).
- Chapters 03 §8 and 11 §6 now list the `NULL-*` IDs their prefix-table row
  assigns to them (NULL-001/003 and NULL-002/003 respectively).
- Chapter 10 §2/§4 dropped the `Parent(...)` builder setter and its default —
  no flavor ever implemented it (parents are wired by containers on add, not
  at build time), so the snippet called a nonexistent API.
- Chapter 20 §4 documents the key-order sensitivity of TypeScript's
  `JSON.stringify` dirty detection (a behavioral probe found
  structurally-equal, key-reordered models read as dirty in TS only;
  FORM-003's guarantee is scoped to same-shaped literals).

## 3. Versioning

Per README §6.1 these are backwards-compatible additions and clarifications:
spec **2.4.0 → 2.5.0**. All four flavors implement the new behavior and bump
minor (2.4.0 → 2.5.0), declaring `min spec = 2.5.0`. The catalog grows from
232 to 234 IDs (229 library + 5 THEME scenario IDs).

> **Forward note (VMX-111).** 234 is the count *after this ADR's* `HIER-018` +
> `NOTIF-017`. The companion **ADR-0038** (same v2.5.0 release) then adds
> `FORM-014`, so the v2.5.0 **end-state catalog total is 235** (230 library +
> 5 THEME scenario IDs) — the figure carried in `spec/README.md` §1.7.

## 4. Alternatives considered

- **Making Swift's lifecycle throwing now.** Rejected for this cycle: it
  cascades `try` through every container hook, builder, sample, and test for
  ergonomic gain only at illegal-use sites; revisit on demand (§2.5).
- **Exempting `Clear()` from Count notification** (the old Python test's
  reading). Rejected: it leaves Count observers stale and contradicts the
  batch-exit rule.
- **A new conformance ID for Clear→Count.** Rejected: §3.3 already covers
  "every mutation that changes Count"; unit tests in all three flavors pin
  the ordering without growing the catalog.
