# ADR 0036 — v2.4.0 platform expansion + theming as a VM concern

**Status:** Accepted (2026-06-02)
**Spec version:** introduced in 2.4.0
**Source:** combined research findings from the four-question post-v2.3 audit (registry publication, Swift feasibility, test coverage, example-app theming gap).

## 1. Context

After v2.3.0 shipped, a five-thread investigation surfaced four loosely-coupled items that all belong in the next release:

1. **Public registries hold nothing.** Despite 23 git tags pushed across the v2.0..v2.3 families, none of the three packages (`VMx` on NuGet, `vmx` on PyPI, `vmx` on npm) has ever been published. The npm name `vmx` is squatted by an unrelated 0.2.1 package owned by `fool2fish`; the original release.yml workflow's never-fired condition has been masking this. The conformance / spec-discipline / typecheck / test gates are excellent but the *distribution* gate doesn't exist.
2. **Swift platform support is missing.** SwiftUI's `ObservableObject` + `@Published` is MVVM-native; Combine is the Apple analog of `System.Reactive` / `rxjs` / `reactivex`. The core VMx primitives (`ComponentVM`, `CompositeVM`, `GroupVM`, `AggregateVMN`, `RelayCommand`, `MessageHub`, `Dispatcher`) all translate naturally.
3. **C# `ForwardingComponentVM<T>` was 71% line covered; `ForwardingCompositeVM<T>` was 8%.** Production infrastructure; property/command/notification delegation through a wrapper is the kind of thing that fails silently. Python aggregate template code was 72%; multi-arity factories (AggregateVM2..6) lacked parametric tests.
4. **Theming lives in views, not in any VM.** Each of the three flagship Notes-Showcase apps hard-codes its palette in framework-native resources (Avalonia ResourceDictionary, Textual TCSS, React CSS custom properties). The pure-VM contract that the apps otherwise enforce strictly is silently relaxed for theme; a consumer who wants a light/dark toggle has no testable seam.

The four items have one thing in common: they are each *additive*. None require breaking any existing surface.

## 2. Decision

Ship them all as a single coordinated v2.4.0 release.

### 2.A — Publication readiness (track A)

- **Rename the TypeScript package to `@thekaveh/vmx`.** The unscoped name is unavailable; the owner-scoped form is the standard npm pattern for owner-published libraries (`@anthropic-ai/sdk`, `@vue/runtime`, etc.). The C# and Python registry names are unchanged (`VMx` / `vmx`).
- **Add `workflow_dispatch` to `.github/workflows/release.yml`** so the workflow can be re-fired for already-pushed tags without retagging. Add `swift-v*` to the tag trigger list (the Swift flavor cuts GitHub Releases, no registry).
- **Configure repository secrets** (`NUGET_API_KEY`, `NPM_TOKEN`) and **PyPI trusted publisher** (OIDC, no secret required) — this is a user-managed step documented in §6 of this ADR.

### 2.B — Test coverage backfill (track B)

- **C# Forwarding decorators to 100% line coverage** via expansion of `tests/VMx.Tests/Forwarding/ForwardingTests.cs`. Covers identity / state / command / lifecycle delegation, hub message forwarding, INPC subscribe & unsubscribe round-trip, collection mutation forwarding, dispose idempotency, and ArgumentNull guards.
- **Python aggregate parametric tests** over arity 1..6 in `langs/python/tests/unit/aggregates/test_aggregate_vm.py`. Covers construction order, dispose cascade, LIFE-013 reconstruct-disposes-prior-slots, per-slot builder validation, BLD-001 setter immutability.
- No production-code changes in track B; this is correctness-floor work only.

### 2.C — Theme as a VM concern (track C)

A new scenario contract at `spec/proposals/2026-06-02-theme-vm-scenario.md`. Implementation per flavor: `ThemeModel` + `ThemeVM` + per-framework `ThemeAdapter`. Five new conformance scenarios `THEME-001..005`. Tests assert the VM-level invariants (command publishes hub message, unknown preset throws, font scale clamps to range, follow-system toggles correctly, high-contrast preserves accent).

ThemeVM is **NOT** wired into `WorkspaceVM` in this release. That requires extending `AggregateVMN` to arity 7 (the next precedent after ADR-0034 added 6). The wiring + arity expansion are a follow-up. The standalone ThemeVM and its adapters are still useful: a flagship app's host page or settings drawer can subscribe to it directly until the arity-7 wiring lands.

### 2.D — Example app edge cases (track D)

The Notes-Showcase flagships gain:
- `InMemoryRepository.failNext(error)` / `failure_mode` constructor option for testable async failure mode.
- Empty-on-startup support (`seed_empty=True`) + a test asserting `WorkspaceVM.construct()` succeeds with zero notebooks.
- Rapid notebook selection concurrency test (verifies the previous load is cancelled when a new selection arrives).
- `NotebookModel.is_readonly` / `isReadonly` field; `CapabilityActionsVM.addNoteCommand.canExecute` becomes false when the current notebook is read-only.

Python + TypeScript flavors land in this release. C# Avalonia is deferred to a follow-up (the Avalonia code-behind audit takes more care; the changes are mechanical but the cycle time is higher).

### 2.E — Swift flavor (track E)

A new `langs/swift/` flavor implementing a subset of the spec. SwiftPM package; minimum Swift 5.9 / iOS 16 / macOS 13 / tvOS 16 / watchOS 9. Combine is the reactive primitive (analog of System.Reactive in C#, rxjs in TS, reactivex in Python). Three preset presets (camelCase per ADR-0006). Conformance coverage in this release: `LIFE-001..013`, `CVM-001..006`, `COMP-001..010` subset, `GRP-001..006` subset, `AGG-001..006`, `CMD-001..007` subset, `BLD-001..005` (47 of 227 IDs).

A follow-up Swift PR will fill in the rest: `HIER-*`, `FORM-*`, `CAP-*`, `DPROP-*`, `COL-*`, `NOTIF-*`, `THR-*`, the `RxDispatcher.default()` equivalent, `ForwardingComponentVM`, all decorator commands, and the SwiftUI Notes-Showcase flagship.

CI on macOS-latest only (Apple's open-source `swift-corelibs-foundation` on Linux does NOT include Combine; cross-platform Swift would need a fallback to `OpenCombine`, which is out of scope).

### 2.F — Spec bump

`spec/VERSION` 2.3.0 → **2.4.0**. Compatibility matrix gains a `swift` column with `2.4.0 (subset)` for the 2.4.x row. All three pre-existing flavor packages bump to 2.4.0 (C# / Python / TypeScript). Companion C# packages (`VMx.Notifications`, `VMx.Extensions.DependencyInjection`) stay at their independent versions per ADR-0009 / 0013.

## 3. Consequences

- New conformance IDs: `THEME-001..005` (5 new), running total 227 → **232**.
- New ADR (0035 → 0036). ADR-0036 supersedes nothing.
- Swift flavor is honestly labelled "subset (2.4.0)" in the compatibility matrix; future PRs grow it toward full parity.
- The release workflow now has a manual escape hatch; users with credential issues can dispatch a single-flavor publish from the GH UI.
- Three coverage gaps are closed in C# + Python; the test count delta is +30 C# + +77 Python (forwarding + aggregates) plus the +9 per flavor edge-case tests in Py + TS + the THEME-001..005 conformance tests.

## 4. User decisions outstanding (after this ships)

These items cannot be self-completed and need user action:

1. **Set repo secrets `NUGET_API_KEY` and `NPM_TOKEN`** in GitHub repo Settings → Secrets and variables → Actions.
2. **Configure PyPI Trusted Publishing** for the `vmx` PyPI project pointing at the `thekaveh/VMx` repo + `release.yml` workflow + the appropriate environment.
3. **Decide AggregateVM7** addition for the next release (needed before `ThemeVM` can be wired into `WorkspaceVM` as a 7th child).
4. **Decide the `examples/csharp/wpf/TodoApp/` and `examples/python/tk/todo_app/` fates** — both are redundant with the Avalonia / Textual flagships; the audit recommends deprecating to a `legacy-examples/` folder.
5. **Decide `examples/python/textual/inspector/` repositioning** — it's a VM-introspection tool, not an example app per se; the audit recommends moving it to `devtools/`.
6. **Decide the cross-platform Swift story** — Linux Swift lacks Combine; if Linux support matters, the path is `OpenCombine` as a fallback. If not, macOS-only CI is correct as shipped.

## 5. Rejected alternatives

- **Negotiate the npm name `vmx` away from `fool2fish`.** Possible but slow; `@thekaveh/vmx` ships now and never costs anything later (the unscoped name can still be acquired in a follow-up release as `vmx` →  metapackage redirecting to `@thekaveh/vmx`).
- **Skip Swift this cycle.** The user asked for it explicitly; the SwiftPM scaffold + 47 IDs is the right "first cut" to land before the SwiftUI flagship work expands the surface.
- **Add `AggregateVM7` + wire `ThemeVM` into `WorkspaceVM` in this PR.** Considered, dropped: the arity expansion is purely additive and easy, but the wiring to all three flagships requires three rounds of test-update work that wouldn't fit tonight's window. Cleaner to ship `ThemeVM` standalone and let the arity-7 + wiring land in a follow-up.
- **Add `ThemeVM` directly to the core library** (under `langs/<flavor>/src/vmx/theme/`). Considered, dropped: theme is *not* a primitive of the MVVM pattern — it's a host-app concern. The library provides the building blocks (`ComponentVM`, `DerivedProperty`, `RelayCommand`); the scenario contract documents how a host app assembles them. This is the same shape as `spec/proposals/2026-05-29-notes-showcase-scenario.md`.
