# ADR 0021 — Post-v2.0 editorial polish (chapter wording, ADR catalogue completeness)

**Status:** Accepted (2026-05-27)
**Spec version:** editorial within 2.0.0 (no normative behavior change)

## 1. Context

A second multi-agent audit, run immediately after the v2.0.0 ship and the
ADR-0020 refresh, surfaced a small cluster of remaining editorial drift in
the spec chapters and ADR catalogue. Each item is purely textual: the
implementations already match the intended behavior; the spec text or the
ADR catalogue had not caught up.

- `spec/00-overview.md` §1 grouped `ForwardingComponentVM` /
  `ForwardingCompositeVM` under "Hierarchical viewmodel types," but per
  `01-concepts.md` and `09-forwarding.md` they are decorators that wrap an
  inner VM, not new tree nodes.
- `spec/01-concepts.md` §1 prose said "VMx defines five viewmodel families"
  while the table beneath it listed six rows (the sixth being the
  forwarding-decorator row).
- `spec/13-tree-utilities.md` §6 listed `UTIL-001..003` and `EXP-001..005`
  as the IDs covering the chapter, but its three bullets only summarised
  the UTIL behaviors — the EXP rows had no matching bullets.
- `spec/12-conformance.md` `NULL-003`'s title read "every core service
  contract," but its Given clause only enumerated `IMessageHub` and
  `IDispatcher`; the sub-package contracts `INotificationHub` (NOTIF-009)
  and `ILocalizer` (LOC-002) have their own null-variant coverage and are
  not in NULL-003's scope.
- `spec/fixtures/derived-properties.json` was missing the
  `"$schema-version": "1.0.0"` header used by the other three fixtures
  (command-truthtable, lifecycle-transitions, message-ordering).
- `spec/ADRs/0001-drop-comscore.md` §4 last bullet said "The Python flavor
  does not yet ship a DI companion … a future `vmx-di` package can be
  added"; per ADR-0009 the absence of a Python DI companion is an
  accepted, deliberate divergence, not a deferral.
- `spec/ADRs/0007-aggregate-vm-arity-1-to-5.md` §1 Context still framed
  the arity decision in the future tense ("We need to decide … for v1.0")
  three minor versions and one major version later; ADR-0020's promise to
  drop v1.0 framing left this one survivor.
- `spec/ADRs/0009-cross-flavor-divergence-catalogue.md` did not record
  the `ILocalizer` shape divergence: C# exposes two overloaded
  `Localize` methods (`Localize(string)` and
  `Localize(string, IEnumerable<object?>)`); Python and TypeScript expose
  a single method with an optional `args` parameter. Without the row, the
  next parity audit would re-discover and re-flag it.

None of these change behavior. They are documentation/catalogue drift.

## 2. Decision

Apply an editorial polish that aligns the spec chapters and ADR catalogue
with what the v2.0.0 implementations already do:

### 2.1 Chapter wording

- `spec/00-overview.md` §1 splits the viewmodel-types bullet so that
  hierarchical types and forwarding decorators are listed separately,
  matching `01-concepts.md`.
- `spec/01-concepts.md` §1 drops the misleading "five" count from the
  prose introducing the viewmodel-types table; the table itself is the
  source of truth.
- `spec/13-tree-utilities.md` §6 adds two bullets summarising the
  EXP-001..005 row: one for `walkExpanded`/`walk_expanded`/`WalkExpanded`
  gating descent on `IExpandable.IsExpanded`, one for the
  `ExpandableState` default/toggle contract.

### 2.2 Conformance catalogue (exempt from ADR-required rule)

- `spec/12-conformance.md` NULL-003 Given clause is tightened to
  enumerate only the base-package core service contracts (`IMessageHub`,
  `IDispatcher`); the sub-package contracts are explicitly noted as
  covered by NOTIF-009 and LOC-002 respectively. No test logic changes.

### 2.3 Fixture schema header

- `spec/fixtures/derived-properties.json` gains the
  `"$schema-version": "1.0.0"` header that the other three fixtures
  already use. The informative `fixture_name`, `schema_version`, and
  `spec_version` keys are dropped; they were not consumed by any
  loader (`grep -rn fixture_name langs/` returns no hits).

### 2.4 ADR catalogue completeness

- `spec/ADRs/0009` gains an `ILocalizer` row in its divergence table:
  C# two overloads, Python/TypeScript one method with optional `args`,
  reason "C# overloading is idiomatic; Python/TS lack method
  overloading."
- `spec/ADRs/0007` §1 rewrites the v1.0-framed Context paragraph in
  past tense and version-agnostic phrasing.
- `spec/ADRs/0001` §4 last bullet rewrites the "does not yet ship a DI
  companion" sentence to record the absence as an accepted divergence
  (cross-referenced to ADR-0009), not a deferral.

## 3. Consequences

- No new conformance ID, no new fixture, no new public API, no behavior
  change in any flavor.
- The conformance-coverage tool keeps reporting 152/152 in all three
  flavors.
- Future parity audits start from a slightly stricter baseline (one more
  row in ADR-0009, one less ambiguity in NULL-003, no surviving v1.0
  prose in chapter or ADR Context sections).

## 4. Rejected alternatives

- **Renumber `01-concepts.md`'s "five" to "six" instead of dropping the
  count.** Six is the *table-row* count but only five rows are
  viewmodel families per `00-overview.md`; the forwarding row lists
  decorators. Dropping the count avoids the ambiguity and is the same
  copy-edit choice already applied to other chapters in ADR-0020.
- **Promote `derived-properties.json`'s richer header (`fixture_name`,
  `schema_version`, `spec_version`) into the other three fixtures.**
  Loaders don't consume those keys, so adding them everywhere would be
  pure noise. Conversely, removing them from `derived-properties.json`
  is mechanically safe (`grep` confirms zero references).
- **Strengthen NULL-003 to enumerate all four contracts (core + sub-
  package).** Would re-do work that NOTIF-009 and LOC-002 already
  perform; only worth doing if a future ID consolidates the four into
  one assertion, which is out of scope for an editorial pass.
