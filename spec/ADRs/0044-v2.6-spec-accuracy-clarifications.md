# ADR 0044 — v2.6 spec-accuracy clarifications (ch. 02 hooks, ch. 15 distinct-until-changed)

**Status:** Accepted (2026-06-21)
**Spec version:** 2.6.0

## 1. Context

A maintenance audit cross-checked the spec against all three full-parity
implementations (C# / Python / TypeScript) and found two places where the spec
text described behavior no flavor implements, or omitted behavior every flavor
implements. Neither is a behavior change — the implementations agree with each
other; only the prose was inaccurate.

1. **Chapter 15 §6 — DerivedProperty distinct-until-changed.** The section
   claimed that on an equal recompute "the new value still replaces `Value`,
   but no `ValueChanged` emission is published," and that this "matches the
   existing `PROP-002` rule for VM properties." In fact all three
   `DerivedProperty` implementations short-circuit on equality **without**
   reassigning (`if (equal) return;`), so the prior instance is retained — and
   the VM property setter the section cites (`_set_model` / `SetModel`) does
   exactly the same (`if (self._model == value) return`). The "still replaces
   `Value`" clause therefore contradicted both the code and the very PROP-002
   rule it invoked. Observable only for equal-but-not-identical objects
   (Python `==`, C# `EqualityComparer.Default`); vacuous in TypeScript
   (`Object.is`). No conformance test exercised it (`DPROP` cases use `int`).

1. **Chapter 02 — subclass lifecycle hooks.** ADR-0041 §3 states the
   `OnConstruct` / `OnDestruct` / `OnDispose` virtual hooks are "already
   published API per `spec/02-lifecycle.md`," but chapter 02 never documented
   them. The hooks exist in `ComponentVMBase` in every flavor and are the
   mechanism by which the container VMs of §6/§7 drive their child cascades, so
   the omission left a real part of the public subclassing surface undocumented
   and made ADR-0041's cross-reference dangle.

## 2. Decision

1. **Chapter 15 §6** is corrected to state that on an equal recompute no
   `ValueChanged` is published **and `Value` retains its current instance** (an
   equal-valued recompute is a no-op), matching the actual code and the PROP-002
   no-op rule.

1. **Chapter 02** gains a new subsection **§2.2 "Subclass lifecycle hooks
   (`OnConstruct` / `OnDestruct` / `OnDispose`)"** documenting the three
   protected overridable hooks, their phase, the per-construct vs. long-lived
   cleanup split (ADR-0041), and their role in the §6/§7 cascades. This backs
   the claim ADR-0041 made.

## 3. Rationale

- The spec is the source of truth, but where the prose contradicts a behavior
  that all flavors implement identically and that a separate accepted ADR
  presupposes, the prose is the defect. Correcting the prose (rather than
  changing three implementations and the VM property setter to match a single
  inaccurate sentence) preserves intended behavior and cross-flavor parity.
- No code changes, no conformance-ID changes, no version bump: these
  clarifications describe existing 2.6.0 behavior accurately.

## 4. Consequences

- A reader of chapter 15 §6 no longer expects an equal recompute to swap the
  stored instance; the documented contract matches PROP-002.
- ADR-0041's "already published API per spec/02-lifecycle.md" reference now
  resolves to spec/02 §2.2. ADR-0041 itself is left unedited (ADRs are
  historical records); this ADR records the follow-through that backs it.
- Optionally, a future `DPROP` conformance case with an equal-but-not-identical
  value type could lock in the retain-prior-instance behavior; not added here.
