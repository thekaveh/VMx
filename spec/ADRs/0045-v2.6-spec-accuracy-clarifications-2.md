# ADR 0045 — v2.6 spec-accuracy clarifications (ch. 06 / 20 / 21)

**Status:** Accepted (2026-06-22)
**Spec version:** 2.6.0

## 1. Context

A follow-up to the audit-report verification pass cross-checked three chapters
against the implementations and found three places where the prose was
self-contradictory or leaked a single flavor's idiom into the language-neutral
shape. None is a behavior change.

1. **Chapter 20 §2 — `FormVM.Model` annotation.** The shape comment read
   `# live, editable (read-only; mutate via SetModel)`, which is
   self-contradictory ("editable (read-only)"). The intent is that `Model` is
   the live working copy, mutated through `SetModel`, not by direct property
   assignment.

1. **Chapter 21 §4.1 — `ObservableDictionary.TryGetValue`.** The signature was
   written `TryGetValue(key1, key2, out value) : bool`, using C#'s `out`
   parameter in an otherwise language-neutral pseudocode chapter. The three
   flavors return the found-flag and value differently (C# `out` param, Python
   a 2-tuple `(bool, value)`, TypeScript a `{ found, value }` object), so the
   neutral shape should not privilege one idiom.

1. **Chapter 06 §3 vs §3.1 — nulling `Current`.** §3 states "Setting
   `Current = null` is always legal," while §3.1's `deselect_component(vm)`
   "raises if `Current != vm`." Read together these appear to conflict; in fact
   they describe two distinct operations (the unconditional raw setter vs. the
   guarded convenience method).

## 2. Decision

1. **Chapter 20 §2** — the `Model` comment becomes
   `# live working copy; mutate via SetModel (no direct assignment)`.

1. **Chapter 21 §4.1** — the signature becomes
   `TryGetValue(key1, key2) : (found, value)` with a comment naming the
   per-flavor shape (C# out-param, Python tuple, TypeScript object), matching
   how the indexer line already annotates per-flavor variation.

1. **Chapter 06 §3.1** — the `deselect_component` bullet gains a clarifying
   sentence: it is a guarded convenience, while the raw `Current = null` setter
   from §3 stays unconditional. No rule changes.

## 3. Rationale

- Same principle as ADR-0044: where the prose is self-contradictory or leaks a
  single flavor's idiom into the language-neutral shape, the prose is the
  defect. Correcting the wording preserves the intended behavior and
  cross-flavor parity.
- No code changes, no conformance-ID changes, no version bump: these
  clarifications describe existing 2.6.0 behavior accurately (cf. ADR-0044).

## 4. Consequences

- `FormVM.Model` is unambiguously documented as a read-only property / live
  working copy mutated via `SetModel`.
- The `ObservableDictionary.TryGetValue` shape no longer reads as C#-only; each
  flavor's return form is named.
- A reader of chapter 06 no longer sees §3 and §3.1 as contradictory: the raw
  setter is unconditional; `deselect_component` is a guarded convenience.
