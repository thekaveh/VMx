# VMx Framework — Comprehensive Critique Report

> Prepared by opencode (z-ai/glm-5.2), 2026-06-27.
> Sourced from deep parallel audits of the spec, all four language flavors, the examples, and the repo docs/tooling/CI. Every claim cites `file:line` in the underlying subagent reports.

## Executive summary

VMx is a disciplined, well-intentioned spec-first MVVM library. The cross-flavor CI gates (spec-discipline, conformance coverage, showcase parity) are real and run on every PR, and the per-flavor test suites are genuinely behavioral rather than smoke tests. **However**, the codebase carries several recurring structural diseases that, taken together, materially raise the cost of using and maintaining the framework:

1. **AggregateVM arity 1..6 is hand-cloned four times** (one per flavor), producing ~4,400 lines of near-identical code that a variadic/tuple design would collapse to a single class.
1. **The builder ceremony is extreme.** Every flagship example spends ~60–70% of its lines on framework wiring (hub, dispatcher, builder chains, subscription bookkeeping, cross-VM message filtering), with no positional-constructor shortcut and no DI-aware builder.
1. **Cross-flavor behavioral divergences are silently breaking ADR-0006's "same shape, only idiom changes" promise** — most seriously Swift's `add()` not emitting `CollectionChanged` (COMP-001 violation) and TS's `senderObject` vs `sender` field split.
1. **The spec is internally inconsistent on counts and stale on several ADRs** — 46 ADRs that include three identical "v2.6 clarifications" (0044/0045/0046), two redundant editorial ADRs (0020/0021), and three "teaching rejections" (0039/0040/0041).
1. **Threading is the weakest chapter** — chapter 11 leaves foreground-emission semantics ambiguous, omits Swift from the dispatcher table, has no await path for Python/TS/Swift background work, and LIFE-008's concurrent-invocation invariant is enforced by nothing.
1. **Real documentation drift:** the entire v2.5.x row of `compatibility-matrix.md` is fabricated (no `v2.5.0` / `spec-v2.5.0` / `*-v2.5.0` tags exist), and "237 verified on every commit" overstates what CI enforces (only 232; the 5 THEME IDs are excluded by design).
1. **Ergonomics gaps that should be framework-owned but aren't:** no `IAsyncCommand`/cancellation, no validation framework, no undo/redo, no UI-framework adapters (every flagship hand-writes a hub→INPC bridge).

What follows is the detailed evidence, organized by theme.

## Theme A — Specification-layer issues

### A1. ADR registry bloat and noise (~9% of ADRs are non-decisions)

- **ADR-0020 and 0021** are both labeled "v2.0 editorial polish, no normative change" (`spec/ADRs/0020-v2.0-spec-text-refresh.md:4`, `0021:4`). They document chapter edits that spec-discipline forced into ADR slots — tautological. Collapse into one or move to a CHANGELOG.
- **ADR-0044 / 0045 / 0046** share the exact title "v2.6 spec-accuracy clarifications," were authored on consecutive days (2026-06-21 / 06-22 / 06-22), all carry "no code change, no conformance-ID change, no version bump" (`0044:55-56`, `0045:52-53`, `0046:36-37`), and follow the identical "audit found prose contradicting code; prose is the defect" template. They are one ADR split into three.
- **ADR-0039 / 0040 / 0041** are "teaching ADRs" that formally close questions already resolved in `proposals/2026-06-13-vmx-absorption-audit-followup.md §6`. Three ADRs for one proposal's rejections.
- **ADR-0036 self-contradicts on the Swift subset** and was never amended: `0036:53` and `:85` still say "53 of 232 IDs," but ADR-0037 (`0037:75-86`) recounts that this was wrong — ~30 citations were shifted/vacuous and the honest count is 39. The README index marks "§2.E corrected by 0037" but the body text was never edited. Readers of 0036 alone get false information.
- **ADR-0009** is 386 lines — by far the longest ADR — mixing a divergence table, 18 named subsections, and a historical appendix. Several subsections (e.g. `TreeStructureChangedMessage` field-name, `0009:48-58`) have no table row. Split into a stable table ADR + per-divergence ADRs.
- **ADR status vocabulary is degenerate.** 44 of 46 ADRs are marked "Accepted"; two carry parenthetical corrections. No "Deprecated / Superseded / Rejected" anywhere. Either no candidate was ever rejected, or rejected proposals were never recorded as ADRs.

### A2. Conformance count arithmetic drifts across versions

The v2.5.0 endpoint is stated three different ways:

- ADR-0037 (`0037:108`): "232 → 234 (229 library + 5 THEME)" — 229+5 = 234 ✓ but contradicts the README.
- ADR-0038 (`0038:58`): "229 → 230 (235 total with THEME)."
- `spec/README.md:136`: "232 → 235 (230 library + 5 THEME)" — internally consistent but contradicts both ADRs.

The Swift subset count is stated three different ways: 53 (ADR-0036, uncorrected), 39 (ADR-0037), 41 (`01-concepts.md:48`). No ADR reconciles 39 → 41.

A `### ` count in `12-conformance.md` returns 238, while the README claims 237. One header is unaccounted for — likely a non-ID `###` (e.g. a "Note" subsection). No machine-validation enforces the count.

### A3. Behavior with no conformance ID

- `06-composite-vm.md:128-132` — `AutoConstructOnAdd(true)` completes the child's transition *before* the `CollectionChanged` event fires; the ordering invariant is not separately testable.
- `15-derived-properties.md:79-80` — post-`Dispose()` `SetValue` is "implementation-defined" with no ID and absent from ADR-0009's divergence catalogue.
- `16-notifications.md:68-73` — duplicate `Post` of the same Notification is "implementation-defined; SHOULD return the existing awaitable" with a "future minor version may strengthen" punt that's been open since v2.0.
- `02-lifecycle.md:142-143` — late-arriving subscriber events on a disposed VM are tolerated with no conformance ID testing the tolerance.
- `08-aggregate-vm.md:55-56` — aggregate raises `PropertyChangedMessage("ComponentN")` on slot population; no `AGG-NNN` pins this.

### A4. Spec vs implementation drift risk (deliberate ambiguity that flavors could diverge on)

- **Child construction order** is described as "unspecified" but immediately pinned by prose: "reference implementations drive them sequentially" (`02-lifecycle.md:127`, `06:119`, `07:36`, `08:50`/`:66`). No conformance ID enforces sequential, so a flavor shipping concurrent construction tomorrow would pass every existing test while breaking subscribers relying on the documented order.
- **`IDialogService` cancellation** is vacuously normative: `19-dialogs.md:99-110` says cancellation "does not throw `OperationCanceledException` unless the host adapter explicitly opts in" — the opt-in escape hatch makes DIA-007 vacuous.
- **`FormVM.OnApproved` divergence** is documented in `0009:314-328` as "deferred to a future release" — C# emits the pre-await captured model; Python/TS emit the live post-await model. Under concurrent `SetModel` during the await (which spec ch.11 permits), the three flavors are observably divergent. FORM-006 only asserts equality in the absence of concurrent re-mutation.
- **Equality / distinct semantics** are per-flavor (`15-derived-properties.md:96-105`: `EqualityComparer<T>.Default` / `==` / `Object.is`) and the fixture `derived-properties.json:36-42` uses primitive integers where all three coincide — the divergence is *untested*. ADR-0044:65 explicitly admits the gap.
- **`ObservableList.batch_update` ref-counting** is normative for `CompositeVM.BatchUpdate` (`06:100`) but the collections chapter (`21:179-196`) never says whether *its* batches are ref-counted.

### A5. Threading chapter (ch.11) is the weakest chapter — 86 lines, largely hand-wavy

- The foreground-emission rule (`11:42-48`) offers two non-equivalent alternatives ("Send from foreground thread" vs "Send on any thread + `ObserveOn(dispatcher.Foreground)`"). Option 1 gives foreground delivery to non-opting subscribers; option 2 gives background. THR-001 only tests the opting-in case, so the divergence is unobserved.
- Background work (`11:51-60`) offers only `ConstructionStatusChangedMessage` subscription as the await mechanism — racy: a subscriber that subscribes *after* background `construct()` reaches `Constructed` never sees the terminal message (per `03-messages.md:82`, "no replay buffer"). **Python/TS/Swift have no async lifecycle at all** (`02-lifecycle.md:31-37` marks async C#-only), so `Background(true)` on those flavors is fire-and-forget with no await path — a serious gap.
- LIFE-008's "MUST raise on concurrent re-invocation" has no specified enforcement primitive (lock/CAS/dispatcher guarantee). Flavors using `Background(true)` with no lock detect the violation only after the state machine observes the wrong `Status`.
- The "default dispatchers" table (`11:24-28`) lists C#, Python, TS — **not Swift**, even though Swift ships `RxDispatcher` per ADR-0036 §2.E.

### A6. Fixtures have no schema and are thinner than the spec implies

- All four fixtures (`lifecycle-transitions.json`, `message-ordering.json`, `command-truthtable.json`, `derived-properties.json`) carry only a `"$schema-version": "1.0.0"` string. No JSON Schema, no `$schema` URI, no validation. A flavor that silently ignores an unknown key is not caught.
- `message-ordering.json:7,14,21,27` uses arbitrary single-character identifiers ("A", "B", "C") as `expected_observed` list members; the fixture doesn't specify the message type, sender, or payload — flavors must symbolically resolve them per-runner.
- `command-truthtable.json:4-8` covers 5 happy/no-op paths but not parameterized commands, predicate raising (`04-commands.md:30-32`), or task raising (`04-commands.md:42`). The chapter claims "five canonical configurations" but the chapter's own prose describes more cases than the fixture exercises.
- `derived-properties.json:30-35` is hardcoded to integers, so the cross-flavor distinct-until-changed divergence is untested. Add an equal-but-not-identical value scenario.

### A7. Spec cohesion and overlap

- `07-group-vm.md` (78 lines) is mostly paraphrase of `06-composite-vm.md` — should be a subsection of ch.06 carrying only the genuine delta.
- `09-forwarding.md:13-16` references `ICompositeVM<VM>` as if canonical, but `06-composite-vm.md` never declares it — `CompositeVM<VM>` implements `IComponentVM + IList<VM> + INotifyCollectionChanged` directly. The interface is introduced in ch.09 and defined nowhere. A reader cannot find its normative definition.
- `SearchableState` / `ExpandableState` are defined in `06:180-213` and `05:127-148` respectively but re-mentioned in `13-tree-utilities.md:62-88`. Their chapter placement is muddled — they belong in `14-capabilities.md` alongside their capability interfaces.
- Broken/stale cross-references: `16-notifications.md:84` self-references instead of pointing at `19-dialogs.md §4`; `13:100` links to ch.18 but ch.18 never links back; `15:139` mentions an "`InitializationTokens` dictionary pattern" with no ADR pointer.

### A8. Missing pieces in the framework

- **Persistence port.** `00-overview.md:79` scopes persistence out, but `FormVM` takes a "persister" delegate (`20-form-vm.md:16-17`) and every flagship invents its own `INoteRepository`/`NoteRepository` (`proposals/2026-05-29...:287-294`). Either formalize an `IPersistence`/`IRepository<T>` port or stop the flagships from reinventing one.
- **Undo/redo beyond command.** `04-commands.md` is fire-only. `HierarchicalVM`'s `TreeStructureChangedMessage` records `Added | Removed | Reparented` but provides no inverse. Conspicuous for a form/tree/CRUD framework.
- **Validation framework.** `FormVM` has `strict` mode (`20-form-vm.md:155-165`) gating on `IsDirty` only — no `IsValid`, no validation-error collection, no `IValidator<T>`. The flagship adds `isValid: DerivedProperty<bool>` ad-hoc (`proposals/2026-05-29...:343`).
- **Async-command cancellation.** `ICommand.Execute() : void` (`04-commands.md:11`) — no `CancellationToken`, no `IAsyncCommand`, no way to cancel a long-running `task`. `IDialogService` has cancellation; commands do not.
- **Accessibility.** `00-overview.md:79` bundles accessibility with rendering as a view concern, but accessible names/roles/labels are VM-level state (`Hint` is already on `IComponentVM` per `01-concepts.md:96`). No `IAccessibility` port, no `AccessibleName`/`AccessibleRole` capability.

### A9. Other design smells in the spec

- **Capability micro-interfaces (ADR-0010) over-fragment.** `14-capabilities.md` lists 22 capabilities including near-tautological triples: `ISelectable`/`IDeselectable`/`ISelectionTogglable`; `IExpandable`/`ICollapsible`/`IExpansionTogglable`; `IConstructable`/`IDestructable`/`IReconstructable` (lifecycle mirrors of ch.02 operations). The CRUD cluster has 7 interfaces (`INewCreatable`, `IDeletable<T>`, `IUpdatable<T>`, `ISavable<T>`, `ICurrentDeletable`, `ICurrentUpdatable`, `IManagable<T>`) for one surface. `IManagable<T>` has one method with no defined semantics. The 22 `CAP-NNN` IDs test interface existence, not behavior.
- **`AggregateVM` arity 1..6** is over-engineered (ADR-0007 rejected variadic generics in C#; ADR-0034 added arity 6 as "purely additive"; the absorption follow-up defers 7–11 with the trigger "a flagship example with ≥7 heterogeneous children" — but `proposals/2026-06-02-theme-vm-scenario.md:114-125` documents that the `WorkspaceVM` arity-7 composition is the *deferred* next step, meaning the trigger has already fired in the flagship that drove arity 6). Either commit to variadic or hold the line at 5; the middle ground (add a class per arity on demand) produces N near-identical copy-paste types per flavor.
- **`FormVM` overlaps with `ComponentVM<M>` + `OnModelChanged`.** `20-form-vm.md:31-41` could be a helper (like `SearchableState`, `ModeledCrudCommands`) instead of a first-class VM type. The asymmetry is unexplained.

### A10. Spec/README and VERSION hygiene

- `spec/README.md:14-162` is 148 lines of per-version release notes before reaching §2 Versioning. The "Contents" section is actually a release-history section. Move to `CHANGELOG.md`, keep `README.md` a flat chapter index.
- `spec/VERSION` is a bare `2.6.0` string with no date, no ADR-pointer, no schema. No `min_spec_version` field.
- `spec/README.md:5-7` embeds per-flavor package versions ("C# 2.6.0, Python 2.6.1, TypeScript 2.6.0, Swift 2.6.0") in the spec README. Any flavor patch bump requires a spec-README edit (which under spec-discipline requires an ADR or a `no-adr-needed` label). Python is already one patch ahead (2.6.1) — the README will drift again.
- `proposals/` is described (`spec/README.md:170-171`) as "historical planning artifacts, not part of the published spec" — but `proposals/2026-06-02-theme-vm-scenario.md` is referenced normatively from `12-conformance.md:2160` and `00-overview.md:54`. The THEME-001..005 conformance IDs live in a "historical" folder.

## Theme B — Cross-cutting implementation smells (all flavors)

These appear in Python, C#, TypeScript, and (where the area is covered) Swift — so they are framework-wide design decisions, not flavor-specific slips.

### B1. AggregateVM arity explosion — hand-cloned four times

| Flavor     | Files                                                   | LOC    | Notes                                                                                               |
| ---------- | ------------------------------------------------------- | ------ | --------------------------------------------------------------------------------------------------- |
| Python     | `aggregates/aggregate_vm.py` + `aggregates/builders.py` | ~1,100 | Six classes + six builders, near-verbatim `_on_construct`/`_on_destruct`/`dispose`                  |
| C#         | `AggregateVM1.cs` … `AggregateVM6.cs` + nested builders | ~1,200 | Six files, linear growth in arity; `OnConstruct` emits N `PropertyChangedMessage`s with no rollback |
| TypeScript | `aggregateVM1.ts` … `aggregateVM6.ts`                   | ~970   | Six files; `aggregateVM6.ts:55-95` is six near-identical blocks                                     |
| Swift      | (subset, but follows same pattern)                      | —      | —                                                                                                   |

Every flavor also ships six builders (`AggregateVM1Builder` … `AggregateVM6Builder`) with hand-written `ComponentN(Func<VMN> factory)` methods. The fix is well-known: tuple-based `AggregateVM<TTuple>` (C#), `AggregateVM[*Vs]` via `TypeVarTuple` (Python 3.11+), or a single `AggregateVM` with a `slots: (() => ComponentVMBase)[]` array and string-keyed accessors (which `tree/walk.ts:40-52` already does internally). The spec's "no variadic generics in C#" rationale (ADR-0007) is from 2012; tuple types and source generators exist now.

### B2. Builder ceremony is the dominant ergonomics problem

Every VM requires the same `.name().services(hub, dispatcher).model(...).build()` chain. Concrete evidence from getting-started docs:

- C#: `ComponentVM<string>.Builder().Name("c").Services(hub, dispatcher).Model("m").Build()` — 4 chained calls + `Build()` for 3 mandatory fields.
- Python: `ComponentVMOf.builder().name("user-card").model(...).services(hub, dispatcher).modeled_hinter(...).build()` — same shape.
- TS: same shape with camelCase.
- Swift: `ComponentVM.builder().name(name).services(hub: h, dispatcher: ImmediateDispatcher.INSTANCE).build()`.

`services(hub, dispatcher)` is required on **every** builder — even when `AddVMx` has already registered them as singletons in DI (`ServiceCollectionExtensions.cs:32-66`). A builder overload resolving from `IServiceProvider` would eliminate the redundancy. A plain positional-options constructor (`new ComponentVM({ name, hub, dispatcher, model })`) would be more idiomatic in TS/Python.

The immutable-copy-on-each-setter pattern is duplicated ~600 lines per flavor (`ComponentVMBuilder`, `CompositeVMBuilder`, `CompositeVMOfBuilder`, `GroupVMBuilder`, `ComponentVMOfBuilder`, `FormVMBuilder`, `HierarchicalVMBuilder`, six `AggregateVMNBuilder`s). The immutability guarantee is asserted in exactly one test (`formVMBuilder.test.ts:70`); builders are virtually always consumed in one fluent chain — premature abstraction.

### B3. Cross-VM hub subscription is the most error-prone pattern in every flagship

Every flagship's `WorkspaceVM` manually subscribes to `notesView.hub.messages` filtered by `sender` + `propertyName` (C# `OfType<>().Where()`, Python `lambda msg: msg.sender is ...`, TS `msg.senderObject === ...`). The same wiring bug was found and fixed three times in parallel ("Round-3 Critical-2" / "Pass-6 real-wiring audit" comments appear in all three flagships). This is a strong signal the framework should provide a typed cross-VM binding helper (e.g. `hub.whenPropertyChanged(sender, "Title").subscribe(...)`).

### B4. No `takeUntil`/`CompositeDisposable` discipline consistency

- **TypeScript** has **zero** uses of `takeUntil`/`takeUntilDestroyed`/`finalize` across the entire codebase. Every subscription is manually pushed to a `Subscription[]` and `unsubscribe()`d in `dispose()`. Non-idiomatic rxjs; more code; more leak surface.
- **C#** uses `CompositeDisposable` correctly in `RelayCommand.cs:26` and `RelayCommandT.cs:28` but `ModeledCrudCommands.cs:23` uses a raw `List<IDisposable>` with a manual `foreach` dispose loop — inconsistent.
- **Python** exposes Subjects as the *same object* via `return self._subject` (`composite_vm.py:103-105`, `group_vm.py:100-103`, `observable_list.py:55-78`), so external callers can `cast` and call `on_next` — leaky abstraction. `MessageHub` does it correctly via `rx.create(self._subscribe_safely)` (`message_hub.py:48-68`).

### B5. Threading / scheduler discipline

- **No `observe_on` anywhere in any flavor's `src/`** (verified by grep on Python and C#; TS uses `queueScheduler` for both foreground *and* background in `RxDispatcher.immediate()` — `services/dispatcher.ts:27-29` — making `asyncSelection` and background scheduling no-ops in every test). The library pushes "remember to `ObserveOn(dispatcher.Foreground)`" onto every consumer; there's no safety net.
- **Background-constructed status transitions are written from the background thread** without marshalling back to foreground: `_set_status` in Python's `base.py:446`, C#'s `ComponentVMBase` background callback (`:267-284`), and TS's `RxDispatcher.immediate()` (which defeats the spec/11 distinction). UI subscribers receive `OnNext` on the background thread unless they remember `observe_on`.
- **`_in_flight` race** (Python `base.py:261-324`, C# `ComponentVMBase.cs:267-284`): the flag is set/cleared without a lock; `Background(True)` schedules on a ThreadPoolScheduler/TaskPoolScheduler thread; a concurrent `destruct()` on the foreground thread reads `_in_flight` without synchronization — classic TOCTOU. Only `NotificationHub` is correctly synchronized (Python uses `threading.RLock`; C# holds `lock(_lock)`).
- **`NotificationHub` publishes `Subject.OnNext` *inside* a lock** (C# `NotificationHub.cs:41`, Python equivalent) — synchronous subscriber dispatch under a lock; any subscriber that re-enters `Post`/`Resolve` stalls or deadlocks.

### B6. Dispose/idempotency inconsistency

- C# sprinkles `GC.SuppressFinalize(this)` on ~6 types (`ComponentVMBase.cs:493`, `CompositeVMBase.cs:344`, all `AggregateVM*.cs`) but **no finalizer exists anywhere in the framework** (grep for `~\w+` returns nothing). The suppression is dead code that implies a finalizer is present. Each call requires a `#pragma warning disable CA1816` to suppress the analyzer that would otherwise complain. Either add finalizers (probably wrong for managed Rx resources) or drop the suppression calls.
- TS sets `#disposed = true` in `decoratorCommand.ts:60`, `compositeCommand.ts:37`, `confirmationDecoratorCommand.ts:51` but **never reads the flag** — `dispose()` is not idempotent in those three classes.
- Python's `ObservableList`/`ObservableDictionary` have **no `dispose()`** at all — their 5 Subjects are never `.on_completed()` or disposed. Long-lived apps that create/discard these collections leak subscriptions.
- C# `MessageHub.Send` (`MessageHub.cs:33-47`) silently swallows `ObjectDisposedException` on a race; pre-check `if (_disposed) return` is non-atomic with `_subject.OnNext`. Documented intent, but a `Debug.Assert` would surface shutdown races in tests.

### B7. `RelayCommand` / `RelayCommandOf<T>` duplication

- TS `commands/relayCommand.ts` ships `RelayCommand` (lines 25-73) and `RelayCommandOf<T>` (118-166) plus their builders — byte-for-byte duplicates modulo the `T` parameter. A single `RelayCommand<T = void>` with overloaded `execute`/`canExecute` would halve the file.
- Python keeps `RelayCommandOf` / `RelayCommandOfT` as aliases (`relay_command.py:277-278`) with an 18-line comment block explaining the dual name and "removal deferred to v3.0.0 per ADR-0009" — deferred indefinitely.
- The "non-generic + generic" split exists in every flavor; the non-generic version is a `RelayCommand<T=void>` in disguise.

### B8. Conformance-test stubs vs. real assertions

- C# spot-checks are real (`HUB-007` sets up a throwing + non-throwing subscriber and asserts; `ThreadingConformanceTests.cs` uses `TestScheduler` with virtual time). Zero `TODO`/`FIXME`/`HACK` across the whole C# tree.
- TS conformance tests are predominantly behavioral (207 `describe("XXX-NNN")` blocks, no empty `it("…", () => {})` stubs found).
- Python has a structural smell: `tests/conformance/test_property_change.py:23-27` and `test_lifecycle.py:153-157` delegate via direct function calls (`from tests.conformance.test_component_vm import test_PROP_001_...; test_PROP_001_...()`) — bypassing pytest fixtures, parametrization, and teardown. A failure inside the delegated function reports under the *delegate's* test ID, not the caller's, confusing the conformance coverage tool. `test_lifecycle.py` has 13 tests but only 8 asserts because 10 of the 13 are one-line delegates that assert nothing locally.
- `tools/check-conformance-coverage.py:85-92` does `pattern.finditer(path.read_text(...))` — pure string-match. A commented-out `# @pytest.mark.conformance("LIFE-001")` satisfies the check. CLAUDE.md acknowledges this but the framing "enforces 100% coverage" overstates what's enforced (it enforces 100% ID-presence, not 100% behavioral coverage).

## Theme C — Per-flavor specific issues

### C1. Python

- **`ForwardingCompositeVM` is a 214-line manual proxy** (`forwarding/composite.py`) — every member of `CompositeVMProto` hand-delegated to `self._wrapped`. Python's `__getattr__` or a `functools.wraps`-style metaclass would shrink to ~30 lines; the explicit surface exists only for mypy but is riddled with 11 `type: ignore` tags (`:177-214`).
- **`_ComponentVMBase` is a god class** (`components/base.py:64-457`, 457 lines) — lifecycle state, selection state, hub, INPC subject, status-trigger Subject, five built-in `RelayCommand`s, selection predicates, parent linkage. The five commands built unconditionally in `__init__` (`:112-146`) are wasted on `GroupVM` (which has no selection slot).
- **`MessageHub._subscribe_safely` swallows all handler exceptions with `except Exception: pass`** (`services/message_hub.py:58-62`) — no logging, no `on_error` propagation. A typo in a subscriber is invisible.
- **`IConstructable.register(ComponentVM)` calls at module import** (`__init__.py:281-289`) — 8 `register()` calls mutate the ABC registry at import time. Side-effecting import.
- **`hierarchical_vm.py:67-68`** defaults `hub` to a fresh `MessageHub()` and `dispatcher` to `RxDispatcher.immediate()` — every other VM requires explicit `services()`. A hierarchical VM silently gets an isolated hub (no parent wiring) unless the user remembers to pass one — a footgun.
- **`__init__.py` exports 127 symbols**, including 12 for aggregates (six classes + six builders), 20 capability micro-interfaces, two aliases for `RelayCommandOf`/`T`, and two ways to get the null hub (`NULL_MESSAGE_HUB` + `null_message_hub_of`).
- **`I`-prefix** on capabilities (`IConstructable`, `IDestructable`, `IManagable`) is a .NET convention, not PEP 8.
- **`component_1` ... `component_6`** 1-indexed property names are non-idiomatic Python.
- **`hierarchical_vm.py:80-82, 160-162`** — `_path_cache` is invalidated by `reparent_child` mutating `child._hierarchical_parent` but the cache (and descendants' caches) are never invalidated. Reading `child.path` after reparent returns a stale list.
- **`pyproject.toml:38-40`** — `force-include` maps `"../../spec/fixtures/lifecycle-transitions.json"` relative to `langs/python/`. Breaks if the package is built from a different cwd. `_load_from_repo` walks the entire parent chain with no depth limit.

### C2. C\#

- **`ComponentVMBase.cs:1-550`** is the framework's core god class — lifecycle state, status-trigger Subject, INPC wiring, **five** built-in commands, background scheduling, in-flight reentrancy guard, selection predicates. The four lifecycle methods (`Construct`/`ConstructAsync`/`Destruct`/`DestructAsync`, lines 249-430) are near-line-for-line clones — should collapse to two parameterized helpers.
- **`SelectNext`/`SelectPrevious`** (`:516-520`) are permanently-disabled commands (`CanSelectNext() => false; SelectNext() {}`) wired with full `RelayCommand.Builder()` ceremony at `:215-225` — five per-instance allocations per `ComponentVMBase` for surface that's always inert. Should be a shared `DisabledCommand.Instance`.
- **22 `IDisposable` types**, each carrying its own `private bool _disposed;` + idempotent-guard idiom repeated ~15 times verbatim. `ModeledCrudCommands.cs:23` uses a raw `List<IDisposable>` instead of `CompositeDisposable` — inconsistent.
- **`default!` lies**: `DerivedProperty.cs:16` (`private TValue _value = default!;`) lies for any reference-type `TValue`. `RelayCommandT.cs:49,62` coerces non-`T` parameters to `default!`, passing `null!` masquerading as non-null to `_predicate`/`_task`.
- **`FormVM.cs:231-233`** uses reflection `MemberwiseClone` with two `!` suppressions; `where TM : notnull` doesn't prevent value types (unbox returns a boxed copy, not the original).
- **~90 `#pragma warning disable`** in `src/` — mostly CA1715/CA1000 (spec-mandated, acceptable) but CA1816 (around the dead `SuppressFinalize` calls) and CA1510 (netstandard2.0 `ArgumentNullException.ThrowIfNull` polyfill) are noise a single `Throw.IfNull` helper would eliminate.
- **`VMx.Extensions/LinqHelpers.cs`** ships `CartesianProduct`, `Sample`, `Product` — unrelated to VMx's domain (ADR-0033 acknowledges C#-only). Move to `VMx.Linq` namespace or drop.
- **`VMx.csproj:30-33`** embeds `..\..\..\..\spec\fixtures\lifecycle-transitions.json` — the 4-level relative path means moving the csproj breaks the build with no compile-time error, only caught at first runtime `Require()`.
- **`Microsoft.Bcl.AsyncInterfaces`** (`Directory.Packages.props:18`) is conditionally included for netstandard2.0 but `rg "using Microsoft.Bcl.AsyncInterfaces"` returns nothing — likely unused dead dependency weight.
- **Test projects target net9.0 only** while the library targets `netstandard2.0;net8.0` — net8.0-specific code paths are exercised only via net9.0 tests.

### C3. TypeScript

- **`hierarchical/hierarchicalVm.ts` is 488 lines** mixing the abstract recursive VM, 13 `as unknown as TVM` recursive-generic casts (lines 113/122/179/182/205/221/243/246/259/262/277/282/298), path-cache invalidation, structural mutation, AND a 137-line builder. Split the builder; reconsider the CRTP-in-TS design (a `HierarchicalNode<TModel>` interface would remove 13 of the 16 total `as unknown as` casts in the codebase).
- **`walk.ts:40-52`** casts `node as unknown as Record<string, unknown>` and probes `component${i}` string keys. If a future `AggregateVM7` adds `component7`, `walk` silently stops traversing it. Stringly-typed reflection.
- **`tests/setup.ts:14`** globally sets `config.onUnhandledError = () => {}` for the whole vitest suite — masks genuine unhandled rxjs errors in any test, not just the HUB-007 case it's documented for.
- **`FormVM.isDirty` uses `JSON.stringify` comparison** (`forms/formVm.ts:126-128`) — breaks for `Date` (serialized to strings), `Map`/`Set` (serialized to `{}`), circular references (throws), `undefined` fields (omitted). The type signature accepts any `TM`; the comment says "for plain objects."
- **`FormVM.approveCommand.execute` fire-and-forgets a promise and swallows rejections** (`formVm.ts:97-107`: `void this.approveAsync().catch(() => undefined)`). A UI bound to `approveCommand` silently swallows persister failures.
- **Committed `src/fixtures/*.json`** with no `// AUTO-GENERATED` marker (JSON can't have comments, but no `.gitattributes` entry marks them as generated). `pretest` auto-syncs them, hiding drift from contributors who commit only the spec change.
- **`RxDispatcher.immediate()`** uses `queueScheduler` for both foreground and background (`services/dispatcher.ts:27-29`), so `asyncSelection: true` and background scheduling are no-ops in every conformance test. The TOCTOU guard at `compositeVMBase.ts:342` is dead code under `immediate()`.
- **`tsup` emits a hashed `relayCommand-Ba0AxAKT.d.cts`** shared chunk even with `splitting: false` — fragile hash name in the tarball.
- **`prepack`/`prebuild` hard-fail outside the monorepo** — `scripts/sync-fixtures.mjs:13` resolves `spec/fixtures` via `__dirname/../../../spec/fixtures`; `npm pack` from a tarball extraction exits 1.
- **`eslint.config.js:46`** disables `@typescript-eslint/no-unsafe-return` globally for a local problem (SENTINEL + copy-constructor builder pattern). A `Sentinel<T>` branded type would fix it locally.
- **`as T` casts from `noUncheckedIndexedAccess`** pepper the collections (`observableDictionary.ts:180,224,322`, `observableList.ts:130,144,169`). An `at(index): T` helper used internally would remove them; the codebase already has `at()` on `CompositeVMBase` (`:141-145`) but doesn't use it internally.

### C4. Swift

- The 41-ID subset is honestly documented per-area (`langs/swift/README.md:124-151`) and tests are real (no stubs). The self-audit in ADR-0037 §2.6 is transparent — this is the most defensible part of the flavor.
- **The hand-rolled transition table is in sync with the fixture by scoping luck, not design.** `ComponentVMBase.swift:405-421` (`_isLegalTransition`) only enumerates `.destructed`/`.constructed` sources; the `default → false` branch happens to match the fixture's illegal-from-transient rows. The `dispose` rows (fixture lines 19-22, 26) aren't represented because `dispose()` bypasses the table. If a future spec change adds a legal `construct`-from-`Constructing` row, Swift silently won't pick it up.
- **`DefaultDispatcher` (`Dispatcher.swift:29-42`) is dead code from the test suite's perspective** — only `ImmediateDispatcher` and `NullDispatcher` are exercised. The `DispatchQueue.global(qos: .userInitiated)` background hop is untested.
- **`ReadonlyComponentVMOf` builder footgun** — Swift can't narrow an inherited `var` setter to unavailable, so `ReadonlyComponentVMOf<T>.builder()` compiles and silently produces a *writable* VM. The dedicated `ReadonlyComponentVMOfBuilder<T>()` exists but the footgun remains.
- **`CompositeVM._setCurrent` traps on a non-child current** (`CompositeVM.swift:134-136`) with no gating `canSetCurrent(_)` predicate. Sibling flavors throw here. Either add a pre-flight or document it as programmer-error-only.
- **`selectNextCommand`/`selectPreviousCommand` are permanently-false predicates with `task: nil`** (`ComponentVMBase.swift:108-113`, `:136-141`) — inert stubs while other flavors actually work when a parent enumerates children.
- **HUB-007 divergence is acknowledged but not mitigated** — `MessageHub.swift:6-13` admits a throwing subscriber would crash the process.

## Theme E — Documentation, tooling, CI, cross-flavor consistency

### E1. CRITICAL — Fabricated v2.5.0 release row

`compatibility-matrix.md:10` and `README.md:159` claim `swift 2.5.0 (subset)` shipped. **There is no `swift-v2.5.0` tag** in the repo (`git tag --list 'swift-v*'` returns only `swift-v2.4.0` and `swift-v2.6.0`). Worse: **no `v2.5.0` repo-wide tag and no `spec-v2.5.0` / `spec-v2.6.0` tags exist** — `v*` jumps from `v2.4.0` to `v2.6.0`; `spec-v*` stops at `spec-v2.4.0`; `csharp-v2.5.0`, `python-v2.5.0`, `typescript-v2.5.0` all don't exist. So the entire 2.5.x row of `compatibility-matrix.md` is fiction — no flavor actually shipped a v2.5.0 artifact, yet `spec/README.md:122-137` describes v2.5.0 as a real spec bump (ADR-0037). This is exactly the gap CONTRIBUTING.md §4.3 warns against ("Do not let source claim X.Y.Z if no vX.Y.Z tag exists").

### E2. "237 verified on every commit" overstates CI

`README.md:12-13` says "237 cross-language conformance IDs verified on every commit." `tools/check-conformance-coverage.py:155-164` reports `Conformance catalog: 232 IDs` and only enforces csharp/python/typescript. The 5 THEME IDs are excluded by `_SCENARIO_PREFIXES` (`:45`). THEME coverage is enforced **nowhere** in CI for the library flavors — the showcase parity check is filename-only (D4). Direction: soften to "232 library conformance IDs verified on every commit; 5 scenario IDs verified by example-app suites."

### E3. Cross-flavor behavioral divergences silently breaking ADR-0006

ADR-0006 promises "same conceptual shape, only idiom changes." Five-spot check of core operations:

1. **Build a Component** — C# is `ComponentVM<T>` (no `Of` suffix); Python/TS/Swift are `ComponentVMOf<M>`. C# deviates; ADR-0006 doesn't address it.
1. **Subscribe to PropertyChanged** — TS exposes `senderObject` (untyped) as the primary field because the typed `sender` is on the generic `PropertyChangedMessage<TSender>` only (`langs/typescript/src/messages/types.ts:10,15`). Python/C# expose a single typed `sender`/`Sender`. `docs/getting-started/typescript.md:99` shows `msg.senderObject === userVM` while `docs/getting-started/python.md:128` shows `msg.sender is user_vm`. Real field-name divergence beyond casing.
1. **Execute a command** — consistent (`execute()`/`canExecute()`), only casing differs.
1. **Add a child to a composite** — **Swift's `add()` does NOT emit `CollectionChanged` and does NOT auto-construct** (`langs/swift/Sources/VMx/Composites/CompositeVM.swift:73-76`), while C#/Python/TS all do. COMP-001 normatively requires "Add emits CollectionChanged(action=Add)". This is a behavioral divergence, not a casing one — the most serious ADR-0006 violation found.
1. **Dispose the hub** — consistent.

There is **no tool that verifies ADR-0006's parity claim**. The §E2 divergences already break it in practice.

### E4. Tooling gaps (no CI gate for)

- **No Swift conformance scraper** (CLAUDE.md acknowledges) — root cause: Swift tests use `/// LIFE-001` doc comments, not a machine-parseable marker. A scraper for doc comments would work; no Swift conformance convention was ever codified.
- **No fixture-sync verification** — TS `sync-fixtures` copies but doesn't verify; Python/C# load at runtime. No CI gate asserts `langs/typescript/src/fixtures/*.json` is byte-identical to `spec/fixtures/*.json`. A 15-line `tools/check-fixture-sync.py` would close this.
- **No compatibility-matrix ↔ git-tag cross-check** — the matrix is hand-maintained and has fabricated rows (E1). A 30-line script asserting every non-`—` cell has a matching `git tag` would have caught it.
- **No `spec/VERSION` ↔ per-flavor `MinSpecVersion` cross-check** — all five match today (verified), but a spec bump that forgets one flavor's declaration would silently slip. A 40-line `tools/check-spec-version-sync.py` would close this.
- **No conformance-ID count citation check** — `README.md` and `spec/README.md` cite "237"/"232" in multiple places, hand-maintained. The scraper prints `Conformance catalog: 232 IDs` but doesn't assert citations match.
- **No API-surface diff tool** — ADR-0006's parity claim is aspirational, not enforced (E3).

### E5. Workflow gaps

- **`release.yml` C# job** (`.github/workflows/release.yml:25-55`) runs `dotnet pack` then `dotnet nuget push` with **no `dotnet test` gate before push** (compare with `python-test` at `:57-100` which runs the full matrix before publication). NuGet uploads are permanent (yank-not-delete), same risk profile as PyPI.

A failed npm publish (partial CDN propagation) wouldn't be caught until a downstream consumer hits it. Also no provenance/attestation (Python has Sigstore; comment at `:266-269` acknowledges the gap).
- **`conformance.yml:71-74`** comment claims "Git pathspec `*` crosses directory separators… `spec/**/*.md` would MISS top-level ones (verified: 41 of 64)." Misleading — modern git globstar matches top-level files in `spec/`. The "41 of 64" stat is stale.

## Closing

The repo is genuinely well-disciplined at the process level — spec-discipline and conformance workflows are real and enforced, per-flavor CI matrices are consistent, and the test suites are behavioral rather than smoke. The recurring diseases are structural: arity-clone duplication, builder ceremony, cross-VM subscription verbosity, and silent cross-flavor divergences that ADR-0006 promises but no tool verifies. The most urgent single fix is reconciling the fabricated v2.5.0 release row; the highest-leverage structural fix is collapsing the `AggregateVM1..6` family across all four flavors.
